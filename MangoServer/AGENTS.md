# MangoServer AGENTS.md

MangoServer 是芒果测试平台 Django 后端。本文件约定后端开发规范，重点覆盖 Django/DRF、数据工厂、MCP 和迁移边界。

## 基本原则

1. 修改前先读已有模型、视图、service、enum，不要凭接口名猜逻辑。
2. 优先沿用现有目录风格：`models.py`、`views/`、`service/`、`urls.py`、`enums/`。
3. 业务逻辑尽量放在 `service/`，ViewSet 只负责入参、权限、序列化和响应。
4. 返回结构使用项目已有 `ResponseData.success/error` 风格。
5. 业务异常优先使用现有 `ToolsError` 或对应模块异常。
6. 不要修改无关模块，不要顺手重构老代码。
7. 遇到用户已有改动，先兼容，不要回滚。

## 命令边界

8. 后端目录是 `MangoServer/`。
9. 常用启动命令：`python start_server.py`。
10. 本地虚拟环境通常是 `.venv/`，Windows 下可用 `.\.venv\Scripts\python.exe`。
11. 开发过程中不允许执行生成迁移脚本命令。
12. 开发过程中不允许执行迁移命令。
13. 迁移脚本由用户自己生成和执行。
14. 如模型发生变化，只说明需要迁移，不要运行 `makemigrations` 或 `migrate`。

## 模型与迁移

15. 修改模型前先确认历史迁移和线上表结构。
16. 新增字段要考虑默认值、历史数据、空值和序列化兼容。
17. 删除字段时同步检查 serializer、service、MCP、前端返回字段。
18. `JSONField` 必须有明确结构约定，不能随意存任意 JSON。
19. JSON 配置建议配合 Pydantic、校验函数或枚举常量定义结构。
20. 外键删除策略优先参考相邻模型，核心数据通常使用 `PROTECT`。
21. 表名沿用 `db_table` 显式命名风格。

## 序列化与查询

22. 列表接口注意 `select_related`、`prefetch_related`，避免 N+1。
23. 详情接口如果返回嵌套对象，应统一 serializer 输出。
24. 搜索字段要和前端查询组件一致。
25. 项目产品、模块字段常见展示依赖嵌套对象，不要只返回 id。
26. 大列表接口谨慎返回重字段和大 JSON。
27. 按模块过滤时考虑跨模块依赖，不能阻断合法关联实体。

## 枚举规范

28. 枚举统一放在 `src/enums/` 或模块既有枚举位置。
29. 枚举值变更要同步前端枚举接口、展示文案和测试。
30. Service 中尽量引用枚举，不要散落数字魔法值。
31. 数据工厂执行状态语义：`0 失败，1 通过，2 待开始，3 进行中`。

## 数据工厂核心约定

32. 工厂实体只描述单表字段规则，不直接绑定具体业务场景。
33. 场景模板是业务入口，可以包含一个主模板和多个关联模板。
34. 后端模型可继续叫 `DataFactoryTemplate`，产品语义是“场景模板”。
35. `DataFactoryTemplateItem` 表示场景里的关联模板。
36. 关联模板默认启用，不再设计 `enabled` 字段。
37. 关联模板清理策略继承场景模板，不再单独维护 `cleanup_strategy`。
38. 关联模板顺序使用 `sort`，后端按 `sort,id` 执行。
39. 执行场景模板时，主模板和关联模板共享同一个运行上下文。
40. 子模板依赖主模板时，应复用上下文，不应重复创建主表。
41. 工厂实体字段规则中的“依赖实体字段”只保存依赖实体和字段。
42. 状态/场景/API 覆盖层才选择具体依赖模板。
43. 依赖字段执行仍以 `template_id` 创建依赖数据。
44. 多表场景必须通过关联模板表达，不要让主表反向自动创建所有子表。

## MCP 规范

45. MCP 工具描述要面向调用者，不要只写后端字段名。
46. 创建/更新场景模板时要支持 `items`。
47. MCP 返回场景模板详情要包含主模板字段和关联模板列表。
48. MCP 应明确工厂实体是单表规则，场景模板是完整业务对象。
49. MCP 入参示例必须和后端校验结构一致。
50. 前端能力变更时，MCP 文案和参数也要同步。

## API 与兼容

51. 已存在接口路径尽量保持兼容，例如 `/data-factory/template`。
52. 产品文案可从“状态模板”改为“场景模板”，内部路径可暂不改。
53. 接口新增字段要兼容老客户端缺省值。
54. 接口移除字段前可在 serializer 入参处忽略旧字段，避免老请求失败。
55. 不要让前端必须一次性加载所有实体、模板、字段。
56. 大数据量场景应按模块、实体懒加载。

## 测试与验证

57. 数据工厂改动至少跑相关 compile/test 或最小测试集。
58. 场景模板测试要覆盖无关联模板、有关联模板、子模板缺字段。
59. 跨模块依赖要作为重要验证场景。
60. 复用上下文要验证不会重复创建主表。
61. 修改枚举后要验证前端展示和后端状态流转。

## 代码风格

62. Python 代码保持项目既有格式，不引入不必要的新框架。
63. 注释只写复杂业务原因，不写显而易见的代码翻译。
64. 查询条件、JSON key、枚举 key 尽量集中定义。
65. Service 方法要命名清楚，避免一个方法同时做预览、执行、清理。
66. 事务边界要明确，执行造数和执行记录更新需考虑失败状态。
67. 记录日志时避免输出敏感 token、Authorization、密码。
68. 最终交付时说明改了哪些文件、是否需要迁移、跑了哪些验证。
